# Project Vision

agentwebui-test 프로젝트의 방향성과 핵심 의사결정을 정리한 문서.

---

## 요약

| 영역 | 내용 |
|------|------|
| **핵심** | 사내 문서 기반 AI 검색 + 지식 기반 사유 응용 (교차 분석, 생성, 추론) |
| **엔진** | ailoy (Rust) — spec/runtime 분리 아키텍처. backend에 라이브러리로 직접 통합 |
| **사용자** | 오픈 플랫폼 방향이나 아직 유동적. test 단계에서는 개발팀 + 주변인 |
| **에이전트 구성** | 비개발자도 설정만으로 만들 수 있는 수준 (모델, 프롬프트, Knowledge) |
| **Knowledge 확장** | 단기: 직접 업로드 문서 → 중기: 클라우드/워크플로우 데이터 연동 |
| **품질 기준** | 추론 품질 + 일관성 + 응용성 + proactive 인사이트. 정확성/관련성은 기본 전제 |
| **배포** | 장기 비전으로 에이전트 공유/배포. 형태는 미정 |

## 1. 이 프로젝트가 해결하려는 문제

### 단기 목표

사내 지식 검색을 넘어서, 문서 기반의 **사유 응용**이 가능한 AI 에이전트 Web UI.

"사유 응용"이란:
- **교차 분석**: Knowledge A의 내용에 다른 기반의 정보(방법론, 인사이트 등)를 적용하여 새로운 결론 도출
- **생성**: 지식을 바탕으로 보고서, 요약, 제안서 작성
- **추론**: 특정 시나리오에 대해 기존 지식 기반 추론
- **Deep Research**: 특정 지식을 기반으로 깊이 있는 조사 수행

### 중기 목표

Knowledge의 소스를 직접 업로드 문서 너머로 확장:
- 클라우드 저장소 (Google Drive, Notion, Confluence 등)
- 워크플로우 상의 정보 (Slack, Jira, 이메일 등)
- 복잡하지 않은 방식으로 이런 다양한 소스를 Knowledge로 묶어서 활용

### 장기 목표

- 특수 AI 에이전트가 유저가 원하는 방향으로 **잘 동작**하게 만드는 것
- 만들어진 에이전트를 **배포**할 수 있는 플랫폼
- 배포 형태(스펙 공유, 호스팅 엔드포인트, 위젯 등)는 아직 열려있음

## 2. 에이전트 품질 기준

### "제대로 동작한다" (기본 전제)

이것은 차별화가 아니라 당연한 조건:
- **정확성**: 문서에 있는 내용을 정확히 찾아서 답한다
- **관련성**: 질문과 관련 없는 문서를 가져오지 않는다

### "잘 동작한다" (핵심 차별화 가치)

이 앱의 에이전트가 추구하는 품질:
- **추론 품질**: 여러 문서를 종합하여 합리적이고 깊이 있는 결론을 도출
- **일관성**: 같은 맥락에서 매번 비슷한 수준의 높은 품질 답변
- **추론 응용성**: 한 종류의 Knowledge에 다른 기반의 정보를 적용하여 새로운 인사이트 생성
- **Proactive 인사이트**: 유저가 묻지 않았지만 필요할 것으로 판단되는 정보를 선제적으로 제공

> 현재 이 품질 기준을 정성적/정량적으로 판단할 수단이 없음. 평가 체계 설계가 필요.

### "잘 보여준다" (부가 가치)

잘 동작한 이후의 문제:
- 응답 형태, 포맷팅, 시각적 구성

## 3. 기술 아키텍처 결정

### 엔진: ailoy

[brekkylab/ailoy](https://github.com/brekkylab/ailoy) — Rust 기반 AI 에이전트 라이브러리.

ailoy의 핵심 구조:
```
ailoy/src/agent/
├── spec.rs    # 에이전트 스펙 (무엇인지 — 모델, 프롬프트, 도구)
├── rt/        # 런타임 (어떻게 실행하는지)
└── mod.rs
```

이 spec/runtime 분리는 에이전트를 스펙으로 정의하고, 그 스펙을 통해 다른 환경에서도 동일한 런타임으로 실행할 수 있게 하려는 설계.

### 통합 방식: A) 라이브러리 직접 호출

```
[Frontend] → HTTP → [Backend (Rust + ailoy)]
                         ↓
                    ailoy crate (같은 프로세스)
                         ↓
                    LLM + RAG 실행
```

- Backend의 `Cargo.toml`에 ailoy를 의존성으로 추가
- 단일 바이너리로 배포 (API 서버 + AI 엔진이 하나의 프로세스)
- 최고 성능, 타입 안전성 확보
- 필요 시 B) 별도 프로세스 분리로 전환 고려 가능

### 에이전트 구성 수준

- **기본**: UI에서 설정만으로 구성 (모델 선택, 시스템 프롬프트, Knowledge 연결)
- **코딩 없음**: 비개발자도 사용 가능한 수준이 기본
- **test 단계**: 도구 선택도 빠진 최소 형태
- **확장 시**: 파워유저를 위한 코드 레벨 커스터마이징 논의 가능

---

## 4. 플랫폼 방향

- **기본 방향**: 오픈 플랫폼 (누구나 에이전트를 만들고 공유)
- **유동적**: 기업용 SaaS 구조도 가능성 열려있음
- **test 단계**: 개발팀 내부 사용 → 주변인 사용성 테스트

---

## 5. 인터뷰에서 드러난 주요 통찰

| 주제 | 통찰 |
|------|------|
| 사유 응용 | 단순 검색이 아닌, Knowledge를 사고의 도구로 활용. 교차 분석이 핵심 |
| Knowledge 확장 | "복잡하지 않게" 가 키워드. 다양한 소스를 쉽게 묶을 수 있어야 함 |
| 품질 정의 | "제대로"와 "잘"의 구분이 명확. Proactive한 인사이트가 차별화 포인트 |
| 평가 체계 부재 | 에이전트 품질을 판단할 수단이 아직 없음. 향후 핵심 과제 |
| ailoy 연동 | 라이브러리 직접 호출(A)로 시작. spec/runtime 분리 활용 |
| 배포 | 비전은 있지만 형태 미정. test 단계에서는 너무 먼 이야기 |
